Opção 1: Job Condicional para Reimportar API

Opção 2: Pipeline Separada para Gerenciamento de APIs

Podemos criar uma pipeline separada focada apenas no gerenciamento de APIs no WSO2, que pode ser acionada manualmente ou por merge requests direcionados a uma branch específica, como api-management.

stages:
  - api-management

check_api_running:
  stage: api-management
  script:
    - ./scripts/check_api_running.sh
  only:
    - api-management

reimport_api:
  stage: api-management
  script:
    - ./scripts/reimport_api.sh
  only:
    - api-management
  needs:
    - check_api_running

Decisão e Racionalização

Faz sentido? Sim, faz sentido ter a capacidade de atualizar a API no WSO2 independentemente do ciclo completo de build e deploy, especialmente se houver casos onde a configuração do WSO2 possa mudar sem alterações no código.

Melhor abordagem: Depende da frequência e do contexto em que essa necessidade surge. Se é uma situação comum, a Opção 2 (pipeline separada) proporciona uma solução limpa e dedicada. Se é algo esporádico, a Opção 1 (jobs manuais) pode ser mais prática e menos complexa.

Implementação recomendada: A Opção 2 oferece uma separação clara de responsabilidades e evita a complexidade de adicionar muitos passos manuais na pipeline principal. Isso facilita o gerenciamento e mantém a pipeline de desenvolvimento mais focada.

Vamos seguir com a opção que melhor se adequa às suas necessidades e fluxo de trabalho. Se precisar de ajustes ou tiver mais perguntas, estou à disposição.

Tem coisas que podem ser adicionadas nesse estágio de develop?

No estágio de develop, além dos passos já mencionados, podemos considerar adicionar outras verificações e processos que garantam a qualidade e a estabilidade do código antes de promover para produção. Aqui estão algumas sugestões:

Adições Recomendadas para a Pipeline de Develop

  1. Testes Automatizados:

    • Unit Tests: Já devem estar inclusos na compilação/build.
    • Integration Tests: Verificar a integração entre diferentes módulos.
    • End-to-End Tests: Simulam o comportamento do usuário final.

  2. Verificação de Segurança:

    • Static Application Security Testing (SAST): Ferramentas como SonarQube, Snyk ou OWASP Dependency-Check.
    • Dynamic Application Security Testing (DAST): Ferramentas como OWASP ZAP ou Burp Suite para testar a aplicação em tempo de execução.

  3. Análise de Cobertura de Código:

    • Geração de relatórios de cobertura de testes para garantir que as mudanças são bem testadas.
    • Ferramentas como JaCoCo para Java.

  4. Verificação de Qualidade de Código:

    • Linter: Ferramentas como Checkstyle, PMD ou SpotBugs para Java.
    • Code Smell e Análise Estática: SonarQube para identificar problemas no código.

  5. Análise de Performance:

    • Testes de carga e performance: Ferramentas como JMeter ou Gatling.

  6. Documentação:

    • Gerar e validar a documentação do código e APIs, utilizando ferramentas como Swagger para APIs REST.
